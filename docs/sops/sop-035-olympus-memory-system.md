# SOP-035-OLYMPUS_MEMORY_SYSTEM

**Status:** ACTIVE  |  **Project:** ZAP-OS / OLYMPUS  |  **Protocol:** B.L.A.S.T.

---

## 1. Context & Purpose

This SOP governs the Olympus Memory v2 system — a biomimetic agent memory layer built entirely on internal infrastructure (MongoDB, ChromaDB, Redis, LangChain, .md files). It replaces flat session checkpoints with structured memory operations: **Retain**, **Recall**, and **Reflect**.

The system is inspired by Hindsight (vectorize-io) concepts but uses zero external dependencies. We own every byte.

---

## 2. Architecture Overview

### 2.1 Memory Types

| Type | Collection | Description |
|------|-----------|-------------|
| **World Facts** | `memory_world` | Rules, patterns, anti-patterns, architecture decisions. The system's knowledge of how things work. |
| **Experiences** | `memory_experiences` | Per-action outcomes from agent sessions. What happened, what worked, what failed. |
| **Mental Models** | `memory_models` | Insights auto-generated by reflect operations. Patterns derived from analyzing experiences. |

### 2.2 Service Roles

| Service | Role | Why |
|---------|------|-----|
| **MongoDB** | Source of truth. Stores all memory documents. | Document store, schema flexibility, aggregation pipelines. |
| **ChromaDB** | Semantic vector search for recall (v2). | Purpose-built for embeddings. Running at `:8100`. |
| **Redis** | Hot cache for domain facts. Pub/sub for cross-agent notifications. | Sub-millisecond reads. Real-time event bus. |
| **LangChain** | Orchestrates reflect operations. | Multi-step LLM chains with our own Gemini/OpenAI keys. |
| **.md files** | Human-readable seed and export format. | Git-trackable, PR-reviewable. |

## 3. Operations

### 3.1 RETAIN (Store)

**When:** End of every task, when a new rule is discovered, when a bug is fixed, when a security incident occurs.

**Mandatory fields for World Facts:**

```json
{
  "category": "mandate | rule | pattern | anti-pattern | architecture | procedure",
  "domain": "layer-system | template | table | typography | security | ...",
  "content": "The fact or rule — clear, concise, actionable",
  "source": "learn.md | session | manual | reflect",
  "tags": ["keyword1", "keyword2"],
  "agent": "spike | jerry | tommy | zeus | system",
  "confidence": 0.0 - 1.0
}
```

**Mandatory fields for Experiences:**

```json
{
  "agent": "spike | jerry | tommy",
  "session_id": "conversation UUID",
  "action": "What the agent did",
  "outcome": "success | failure | partial",
  "context": "Why this was needed",
  "lesson": "What was learned",
  "files_touched": ["file1.tsx", "file2.tsx"],
  "domain": "domain tag",
  "tags": ["keyword1", "keyword2"]
}
```

### 3.2 RECALL (Retrieve)

**When:** Start of every session, before working on a domain, before implementing a pattern.

**v1 — Exact Recall (Tag-based):**

```javascript
// Recall all layer-system rules
db.memory_world.find({ domain: "layer-system" })
  .sort({ confidence: -1, updated_at: -1 })
  .limit(10);

// Recall Spike's recent experiences with templates
db.memory_experiences.find({ agent: "spike", domain: "template" })
  .sort({ timestamp: -1 })
  .limit(10);
```

**v2 — Semantic Recall (ChromaDB):**

```javascript
// Find memories similar to a query, regardless of exact tags
chromadb.query({
  collection: "memory_vectors",
  query_texts: ["How should table headers be colored?"],
  n_results: 5
});
```

### 3.3 REFLECT (Analyze)

**When:** End of milestone, weekly automated, on-demand by Zeus.

**Process:**
1. Pull recent experiences from `memory_experiences` (last N days)
2. Pull relevant world facts from `memory_world`
3. Send to Gemini via LangChain with structured prompt
4. Parse response into insight documents
5. Insert into `memory_models`
6. High-confidence insights (≥ 0.9) auto-promoted to `memory_world`

## 4. Memory Layers

| Layer | Timeframe | Service | Purpose |
|-------|-----------|---------|---------|
| **Short-Term** | Current session | Redis | Hot cache of active context + recent domain facts |
| **Mid-Term** | Cross-session (days/weeks) | MongoDB | Granular experiences with tag + text search recall |
| **Long-Term** | Institutional (months/forever) | MongoDB + ChromaDB | Self-updating world facts + auto-generated mental models |

---

## 5. Self-Teaching Loop

```
Agent Works → RETAIN experience → REFLECT finds patterns →
High-confidence insight promoted to World Fact →
Next RECALL returns new fact → Agent behavior changes →
New experiences generated → LOOP
```

No human in the loop for this cycle. The system genuinely learns from its own work.

---

## 6. learn.md Relationship

| Role | Description |
|------|-------------|
| **Seed file** | On first boot or reset, parse `learn.md` and retain each section as a world fact |
| **Export target** | Periodically generate `learn.md` from `memory_world` for human review |
| **Git tracking** | `learn.md` stays in the repo — reviewable in PRs, readable by humans |
| **NOT the runtime source** | Agents recall from MongoDB/Redis at runtime, not from the .md file |

---

## 7. Security Memory Bank

World facts with `domain: "security"` form the security memory bank:

- **Retain** known injection patterns, XSS signatures, MCP poisoning patterns
- **Recall** against the security bank before processing untrusted input
- **Reflect** on security experiences to identify recurring vulnerability patterns

**Two-layer defense:**
1. **Redis** — Cached exact pattern match (< 1ms, known threats)
2. **ChromaDB** — Semantic similarity search (< 50ms, novel variants)

## 8. Database Indexes

### memory_world
- `{ domain: 1, category: 1, confidence: -1 }` — Primary query path
- `{ tags: 1 }` — Tag-based filtering
- `{ content: "text" }` — Full-text search

### memory_experiences
- `{ agent: 1, domain: 1, timestamp: -1 }` — Agent + domain + recency
- `{ tags: 1 }` — Tag-based filtering
- `{ outcome: 1 }` — Filter by success/failure
- `{ action: "text", lesson: "text", context: "text" }` — Full-text search

### memory_models
- `{ domain: 1, confidence: -1, created_at: -1 }` — Domain + quality + recency
- `{ insight: "text" }` — Full-text search

## 10. Non-Negotiable Rules

1. **Zero external memory dependencies.** No third-party packages manage our agent memory. Ever.
2. **Every retain must have a domain tag.** Untagged memories are unsearchable.
3. **Experiences are granular.** One action = one experience document. No monolithic session dumps.
4. **Confidence scores are mandatory.** World facts and models must declare their confidence level.
5. **learn.md stays in git.** It's the human-readable contract. MongoDB is the runtime source.
6. **Security patterns are always-warm in Redis.** Never cold-start the security layer.
7. **Reflect output is never auto-deployed.** High-confidence insights are promoted to world facts, but architectural changes require Zeus approval.
8. **No agent ships without memory.** Every new agent must complete the Agent Genesis Checklist (Section 11) before deployment.

## 11. Agent Genesis Checklist

When spawning a new agent, memory provisioning is **not optional**. It is step 1 of the agent's existence. An agent without memory cannot learn, cannot recall, and cannot participate in the self-teaching loop.

### 11.1 Pre-Spawn Template

Every new agent must have these resolved before the first task:

| # | Step | Command / Action | Verification |
|---|------|-----------------|--------------|
| 1 | **Name the agent** | Assign a unique agent ID (e.g., `spike`, `jerry`, `tommy`, `ralph`) | ID is lowercase, no spaces, no special chars |
| 2 | **Assign domain(s)** | Declare primary domains the agent operates in (e.g., `zap-design`, `fleet`, `security`) | Domain tags exist in `memory_world` taxonomy |
| 3 | **Create memory bank** | Insert agent registration into `memory_world` | `db.memory_world.findOne({ agent: "new-agent" })` returns doc |
| 4 | **Seed world facts** | Copy shared world facts (mandates, architecture rules) into agent's recall scope | Agent can recall MANDATE-001 |
| 5 | **Seed domain facts** | Retain domain-specific rules for the agent's area | Agent can recall its domain rules |
| 6 | **Register identity** | Add agent to `SYS_OS_identity_directory` collection | Agent appears in identity lookup |
| 7 | **Verify recall** | Run test recall query against agent's domain | Returns ≥ 1 relevant world fact |
| 8 | **Subscribe to pub/sub** | Wire agent to Redis `memory:retained` channel | Agent receives cross-agent retain events |
| 9 | **First retain test** | Agent retains a test experience | `db.memory_experiences.findOne({ agent: "new-agent" })` returns doc |
| 10 | **Connect to reflect pipeline** | Ensure agent's experiences are included in scheduled reflect runs | Agent domain appears in reflect aggregation query |

### 11.4 Post-Spawn Validation Script

Run this after completing the checklist to verify the agent is memory-ready:

```javascript
// Validate agent memory provisioning
const agentName = "NEW_AGENT_NAME";

// 1. Agent registration exists
const reg = await db.memory_world.findOne({ 
  category: "agent-registration", 
  agent: agentName 
});
assert(reg, "Agent registration missing");

// 2. Agent can recall mandates
const mandates = await db.memory_world.find({ 
  category: "mandate" 
}).toArray();
assert(mandates.length > 0, "No mandates found — shared world facts not seeded");

// 3. Agent can retain
await db.memory_experiences.insertOne({
  agent: agentName,
  action: "genesis-validation",
  outcome: "success",
  context: "Post-spawn validation test",
  lesson: "Agent memory system is operational",
  domain: "agent-genesis",
  tags: ["genesis", "validation"],
  timestamp: new Date()
});

// 4. Agent experience persisted
const exp = await db.memory_experiences.findOne({ 
  agent: agentName, 
  action: "genesis-validation" 
});
assert(exp, "Agent cannot retain experiences");

console.log(`✅ Agent '${agentName}' memory provisioning validated.`);
```
