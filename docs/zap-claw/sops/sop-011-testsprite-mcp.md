# SOP-011: TestSprite MCP & Verification Handshaking

## 1. Purpose

This Standard Operating Procedure formalizes how the Claude Agent Team, Antigravity, and Jerry (The Watchdog) interface with the **TestSprite Model Context Protocol (MCP)** server. It defines the exact trigger conditions for autonomous QA loops to prevent hallucinated code deployments.

## 2. When to Use TestSprite

**Do NOT use TestSprite for:**

* Syntax or linting errors (use `tsc` or `Eslint`).
* Database schema validation (use Prisma CLI).
* Cosmetic updates (unless tied to visual regression).

**MANDATORY TestSprite Triggers:**

* **PR/Merge Gate:** When Claude completes a full feature branch (e.g., Auth UI, User API) and is ready to mark it "Done."
* **Refactoring:** Whenever a core `src/runtime` or Next.js layout file is modified.
* **Hydra Handoff:** When executing a PRD generated by Antigravity under the 3-Way Hydra loop, Claude MUST hand the task off to Jerry, who will invoke TestSprite before merging to main.

## 3. How to Execute

### For Zeus (Local Machine Configuration)

To physically connect the agent workforce to TestSprite, Zeus must add the server block to the IDE's MCP Configuration file (Cursor, VSCode extension, or ZAP Agent configs):

```json
{
  "mcpServers": {
    "TestSprite": {
      "command": "npx",
      "args": ["-y", "@testsprite/testsprite-mcp@latest"],
      "env": {
        "API_KEY": "<ZEUS_INSERT_TESTSPRITE_KEY_HERE>"
      }
    }
  }
}
```

### For Claude & Agent Workforce (The Testing Commands)

Agents do not manually write Selenium or Puppeteer tests anymore. When Claude finishes a functional block, it must explicitly output the following trigger prompt to the Watchdog (or invoke the tool directly if operating autonomously):

> "Help me test this project with TestSprite"

* **Sub-Scoping:** You can constrain TestSprite by specifying the target: *"Help me test the /api/hud/chat endpoint in this project with TestSprite."*
* **Execution Location:** TestSprite tests are executed in the secure cloud, abstracting away local headless browser requirements.

## 4. Error Resolution (The TRT Feedback Loop)

When TestSprite returns a failing output, the agent MUST NOT blindly try to "guess" the fix. The agent must adhere strictly to **SOP-010 (Test-Time Recursive Thinking)**.

1. **Capture the Cloud Trace:** Read the exact error report sent back by the TestSprite MCP.
2. **Generate [Error Report]:** Before writing new code, output an `[Error Report]` block explaining the exact reasoning flaw that caused the crash.
3. **Appended Knowledge:** Add the failed attempt to the `knowledge_text` block (e.g., `## Wrong Answer List`) to ensure the agent's context window recursively avoids the failed logical path.
4. **Rewrite and Retrigger:** Re-run the TestSprite MCP after deploying the fix. Do not mark the task complete until TestSprite exits with a verified success.

## 5. Summary Hierarchy

1. **Antigravity:** Writes the blueprint.
2. **Claude:** Writes the code.
3. **Jerry:** Says: "Help me test this project with TestSprite".
4. **TestSprite:** Executes cloud tests. Jerry receives Pass/Fail.
5. **TRT Loop:** Failures trigger recursive self-analysis (SOP-010) until success.
